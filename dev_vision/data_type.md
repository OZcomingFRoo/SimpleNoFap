# Data Types

The app will use 2 different data storage layers:

- Room/SQLite database for structured app records
- Jetpack DataStore Preferences for lightweight user settings

Room DAOs are a persistence detail. App code should access structured records through repositories, not DAOs directly. Validation and business rules belong in repositories or domain helpers.

## User

This will be set in Jetpack DataStore Preferences:

- Name (String)
- Selected Theme required (Enum value of: Light (Default), Dark, and based on System)
- Selected Language required (Enum value of: English (Default), Hebrew, and based on System's language)
- StartNoFapAt required (Datetime  that records the last relapse the user did)

The value "StartNoFapAt" is the crurial part of the app

## Schedule Notifications

This will be a Room database table that keeps all scheduled notifications.
The purpose of the notifications is make the user aware to not give in to the urge at hours where he's most susceptible to relapse.
Words of encouragement and warning should accompany the user.

Columns:

- id required (autogenerated primary key)
- createdAtEpochMillis required
- updatedAtEpochMillis required
- daysOfWeekMask required (int bitmask: Sunday=1, Monday=2, Tuesday=4, Wednesday=8, Thursday=16, Friday=32, Saturday=64; valid range 1..127)
- timeMinutesOfDay required (int, valid range 0..1439)
- messagePresetKey optional (app-owned localization key for preset message)
- customMessage optional
- titlePresetKey optional (app-owned localization key for preset title)
- customTitle optional
- Active (Boolean for turning on/off the notification)

Do not store localized notification text or icon assets in this table. Resolve preset text and visual choices from app-owned mappings and localization.

## DayStreak (rewards that can be used)

This will be a Room database table.
These reward play a crurial part for encouraging users to keep going, and not give in easily to one relapse.
Some users can hold off for 3 weeks, but might fail once.
Instead of simply resetting the counter to ZERO, let users use "DayStreak" to hold the reset until they manage to hold off or not.
The point is not to force users to reset on every small mistake but encourage them to slowly stop succumbing to the urge, as a smoker needs to smoke less and less as time goes on until he stops completely.

So, if a user manages to for 3 weeks to urge and then he does, let him use the "DaySteak" to save himself from resetting.
If the user has no un-used DayStreaks, or he no option but to reset.

Note - this database is managed by the system when it comes to creating and deleting.
The user obviously have limited options to manage it

Colums:

 - id required (autogenerated primary key)
 - createdAtEpochMillis required
 - updatedAtEpochMillis required
 - streakType required (string value from the app-owned DayStreakType enum)
 - achievedAtEpochMillis required
 - usedAtEpochMillis optional (null means unused/available)
 - sourceStreakStartAtEpochMillis optional

Do not store localized Day-Streak names or icons in this table. Resolve Day-Streak names, required day counts, and icon keys from app-owned metadata mappings.
